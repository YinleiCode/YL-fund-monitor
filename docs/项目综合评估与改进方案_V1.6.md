## 2. P0 — 致命问题（立即修复）

### 2.1 [P0] `fetch_sh_index_change()` 接口失败返回 0.0 → 情绪分虚假

**文件：** `data_fetcher.py` 第 893-914 行 → `market_guard.py` 第 35-42 行

**问题：** 当 akshare 接口失败时返回 `0.0`，这和"上证指数今日收平"在语义上完全无法区分。`market_guard.calc_sentiment()` 把 `0.0` 当作真实数据使用，导致：
- 真实大跌时接口失败 → 情绪分虚高，系统错误允许买入
- 真实大涨时接口失败 → 情绪分被低估，系统错误否决买入

**2026-05-28 真实案例：** 上证指数接口全崩 → 返回 0.0 → 情绪分 9.0/10 → 系统认为"积极行情" → 实际行情可能是下跌的。

**解决方案：**

`data_fetcher.py` 中修改：
```python
def fetch_sh_index_change(trade_date: str) -> Optional[float]:
    """返回 Optional[float]，失败时返回 None"""
    ...
    return None   # 接口失败，不再返回 0.0
```

`market_guard.py` 中修改：
```python
def calc_sentiment(..., index_change_pct: Optional[float], ...) -> dict:
    if index_change_pct is None:
        # 数据缺失：情绪分设定为"数据不可用"状态
        return {
            "score": -1,           # -1 表示数据不可用
            "detail": {"index_status": "API_FAILED"},
            "strategy": "数据缺失，仅观察",
        }
```

`trade_review.py` 中改动 `check_buy()` 的 `market_sentiment_below_5` 判定：
- 当情绪分为 -1（不可用）时，**不自动否决买入**，但日志中标记"情绪数据缺失"
- 这比 0.0 的虚假低分或虚假高分都好

### 2.2 [P0] build_market_daily.py 文件缺失

**问题：** 策略说明书 V1.6 和代码审计报告都引用了 `build_market_daily.py`，但该文件实际**不存在于项目目录**。这导致：
- `output/market_daily/` 目录无法生成
- `build_tomorrow_plan.py` 依赖的 market_daily 输入始终为空
- V1.6 "赚钱效应快照"成为空壳，`market_env_verdict` 永远输出"数据不足"

**现状补救：** 立即在 `docs/V1.6_market_breadth_cache_design.md` 基础上实现派生脚本。

**解决方案（三步走）：**
1. **紧急（今天）**：把 `market_guard.calc_sentiment()` 的输出直接写成 `market_daily_YYYYMMDD.csv`，至少不空
2. **短期（本周）**：实现 `scripts/build_market_breadth_cache.py` 抓真实涨停/跌停/涨跌家数
3. **中期（两周）**：打通 `build_market_daily.py` → `build_tomorrow_plan.py` 的整条处理链路

## 3. P1 — 严重问题（今日修复）

### 3.1 [P1] V1.6 计划日期不匹配 → 计划层全部失效

**日志证据：**
```
2026-05-28 08:56:31,217 [WARNING] trade_review:
  [v16] plan.next_trade_date='20260529' != report_date='20260528'
```

**问题：** `build_tomorrow_plan.py` 生成的 plan 中 `next_trade_date` 是 20260529，但 5/28 盘前运行的 report_date 是 20260528。两边的日期算法不一致，导致整个 V1.6 计划层对当天的推荐票全部 fallback。

**根因追溯：** `build_tomorrow_plan.py` 的日期算法与 `data_fetcher.calc_dates()` 不一致。前者在周五生成下周一计划时，`next_trade_date` 用的 `next_trading_date()` 排除周末；后者在 `calc_dates()` 中的逻辑依赖于 `now.hour >= 15` 判断，两者跳转逻辑不完全对齐。

**解决方案：**
1. 统一两边的交易日推进逻辑，共用 `data_fetcher.next_trading_date()`
2. 在 `build_tomorrow_plan.py` 中加入测试：跑完计划后打印 `plan.next_trade_date`，人工验证与 `data_fetcher.calc_dates()[1]` 一致
3. 在 `_load_v16_plan_for_report_date()` 中增加宽松匹配：if `plan.next_trade_date` in [`report_date`, `next_trading_date(report_date)`] 都视为匹配

### 3.2 [P1] 主题龙头模式 API 全崩 → 完全依赖过期缓存

**日志证据（5/28）：**
```
全部 21 个板块的成分股 API 全部 RemoteDisconnected
"商业航天"、"卫星互联网"、"航天航空"、"机器人执行器"等 5 个板块
连磁盘缓存都没有 → 记为失败
剩余 16 个板块使用前一天的成分股缓存
```

**问题：** `theme_auto.py` 对东方财富 API 的单点依赖太强。一旦接口挂掉，整个模式退化为读取前一日缓存，板块行情可能是过时的。

**解决方案：**
1. **短期间隔分批请求**：不要一起发 21 个请求，加 0.5-1 秒的间隔减少触发限流
2. **增加同花顺板块数据作为第二数据源 fallback**（与 money_flow 共享 fallback 策略）
3. **缓存有效期延长**：成分股变化慢，可接受 3-5 日内的缓存；`theme_auto.py` 当前缓存日期判断过于严格

### 3.3 [P1] 历史K线全走新浪 → 缺换手率、量比

**2026-05-28 数据：** 98/98 只股票历史K线全部来自新浪，东方财富 0 只。新浪 `stock_zh_a_daily()` 不返回换手率、量比等关键字段。

**影响：** 
- `filters.py` 中换手率过滤用的是 `df["turnover_rate"].isna() | (df["turnover_rate"] >= min)` → 新浪数据导致换手率全为 NaN，所有候选通过粗筛
- `scorer.py` 的成交量分位、换手率健康度打分失去意义
- `indicators.py` 的 `vol_ratio`（量比）计算用的是成交量/20日均量（不依赖换手率字段），所以部分指标还 OK，但换手率相关全部无效

**解决方案：** 
1. 短期内：在 `_process_spot_raw()` 中对新浪数据增加警告日志
2. 长期：在 `prefer_source: auto` 模式下，东方财富即使延迟也要优先尝试

### 3.4 [P1] V1.6 三层买入架构中"赚钱效应快照"层完全空跑

**问题：** 策略说明书 V1.6 描述的"复盘计划层"需要 `market_state` / `sector_data_status` 等输入，这些来自于 `build_market_daily.py`。该文件缺失导致：
- `trade_permission` 无法准确判断
- `allowed_themes` 缺少数据支撑
- 复盘计划层的"数据驱动"变成空架子

**当前实际运行状态：**
```
复盘计划层：V1.6 标签打了，但数据来源（market_daily）缺失 → 部分降级运行
资金条件层：observe_only 模式，记录但不拦截 → 最稳定的层
9:36技术确认层：全量运行 → 当前实际决策主力
```

**解决方案：** 与 2.2 相同的三步走策略。

### 3.5 [P1] config.yaml `version_effective_from: "20260529"` 与文档冲突

**现状：** 
- `config.yaml` 中：`version_effective_from: "20260529"`
- `docs/朱哥A股短线雷达_策略说明书.md` 中：「9:36 技术确认层切换日期：`config.yaml → buy_rules.version_effective_from = 20260527`」

## 4. P2 — 中等问题（本周修复）

### 4.1 [P2] 北交所涨跌幅规则遗漏

**影响范围：** `filters.py` 的 `_limit_ratio()` + `trade_review.py` 的一字涨停判定

**问题：** 北交所（4/8 开头股票）涨跌幅限制为 30%，当前代码只处理了创业板（300/301）的 20% 和 科创板（688）的 20%，4/8 开头股票被按主板 10% 处理。

**解决方案：** 统一修改两处的 `_limit_ratio()`：

```python
def _limit_ratio(code: str) -> float:
    code = str(code).zfill(6)
    if code.startswith(("300", "301", "688")):
        return 0.20
    if code.startswith(("4", "8")):
        return 0.30
    return 0.10
```

### 4.2 [P2] 一字涨停股价冻结判定用绝对值而非百分比

**文件：** `trade_review.py` 第 1022-1034 行
```python
price_frozen = abs(cur_price - open_p) < 0.01
```

**问题：** 对高价股（>200元），正常波动 0.01 元会被判为"价格冻结"。应改为百分比：
```python
price_frozen = open_p > 0 and abs((cur_price - open_p) / open_p) < 0.001
```

### 4.3 [P2] 买入滑点不对称

**现状：** 
- 买入：总是扣滑点（价格上浮 0.1%）
- 卖出：`apply_sell_slippage: false`，永不扣滑点

**影响：** 模拟收益系统性地被低估。每次买入模拟成本比实际多 0.1%，但卖出时不抵消。平均每笔交易亏损 0.1% * 2 = 0.2% 的统计偏差。

**建议：** 在卖出端也启用滑点（对称处理），或从买入端去掉滑点只用真实价格。对称处理更合理：
```yaml
apply_sell_slippage: true
sell_slippage: 0.001
```

### 4.4 [P2] 同花顺 fallback 20 秒阻塞 9:36 窗口

**文件：** `money_flow.py` 第 321-384 行

**问题：** `_fetch_ths_snapshot()` 一次拉全市场需要 ~20 秒。如果在 9:36 买入窗口期间触发 fallback，将阻塞 20 秒，影响后续股票的行情时效性。

**建议：** 把 THS 快照的加载提前到盘前（08:55 与主题龙头同步），通过提前缓存避免盘中阻塞。

### 4.5 [P2] V1.6 日志重复问题

**文件：** `trade_review.py` 第 621-635 行

**问题：** 当日计划缺失时，6 只候选票各自输出一行 "明日计划缺失或日期不匹配" 的警告，日志被污染。

**建议：** 在加载后立即决定是否 fallback，出一行日志，然后让 `_apply_v16_plan_to_candidate` 静默 fallback。

### 4.6 [P2] `dashboard_app.py` 6500 行 — 严重违反单一职责

**文件：** `dashboard_app.py` 代码体积（实际已超 6500 行）

**问题：** 包含数据加载、6 个独立页面、V1.6 资金源自检、UI 组件、工具函数等，修任何一个功能都要翻整个文件。

**建议：** 拆分为 `dashboard/` 包：
- `dashboard/main.py` — 主入口 + 路由
- `dashboard/pages.py` — 各 page_* 函数
- `dashboard/components.py` — UI 组件
- `dashboard/data.py` — 数据加载转换
- `dashboard/utils.py` — 工具函数

## 6. 系统性改进方案

### 6.1 数据源可靠性体系

当前系统依赖 5 个免费数据接口，其中 3 个（东方财富 push2his、概念板块、成分股）在 5/28 全部不可用。

**建议架构：**

```
                ┌─────────────────┐
                │  数据源管理器    │
                │  DataSourceMgr  │
                └────────┬────────┘
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ 东方财富  │  │  新浪    │  │  同花顺  │
    │ (primary) │  │(fallback1)│  │(fallback2)│
    └──────────┘  └──────────┘  └──────────┘
          │              │              │
          ▼              ▼              ▼
    ┌──────────────────────────────────────┐
    │       健康探针 (Health Probe)         │
    │  - 大市值探针 (检测接口连通性)        │
    │  - 中小市值探针 (检测数据完整性)      │
    │  - 新鲜度探针 (检测数据是否为冻结)    │
    └──────────────────────────────────────┘
```

**具体操作：**

1. **健康探针前置化**：在每天 08:50 先运行健康探针，判断各个数据源的状态。不再等到盘中才知道接口挂了。
2. **数据源优先级动态调整**：如果东方财富连续 2 天不可用，自动降级为以新浪为主、同花顺为备。
3. **缓存策略升级**：
   - 行情快照缓存保留 5 天（当前已实现）
   - 板块成分股缓存保留 7 天（成分股变化慢）
   - 资金流缓存保留 1 天（资金流变化快）
   - 所有缓存带新鲜度时间戳和校验和

### 6.2 日期一致性体系

当前系统有 3 个不同的日期逻辑：

| 模块 | 日期算法 | 用途 |
|------|---------|------|
| `data_fetcher.calc_dates()` | 根据 `now.hour >= 15` 判断 | 全系统 |
| `build_tomorrow_plan.py` | 自己的日期跳转逻辑 | V1.6 计划 |
| `auto_supervisor.py` | 自己的日期判断 | 任务调度 |

三者不完全对齐，导致 V1.6 计划日期不匹配。

**建议：** 
1. 建立一个 `trading_calendar.py` 模块，封装所有日期计算
2. 所有模块从此导入 `calc_dates()` / `next_trading_date()` / `prev_trading_date()`
3. 添加一个 `--check-dates` 参数用于调试日期对齐

├── run.py                   # 主入口（精简后）
├── auto_supervisor.py       # supervisor
├── scripts/                 # 派生脚本

```

**迁移原则：**
- 保持 `run.py` / `auto_supervisor.py` 在根目录作为入口
- `src/` 下通过 `from src.xxx import yyy` 引用
- 一次迁移一个模块，每模块确认 1-2 个交易日正常运行后再迁移下一个
- 不修改 CSV 的输出格式和字段名

### 6.4 监控与告警体系

**当前缺失：** 没有任何"数据质量监控"。5/28 所有 API 挂掉时，系统静默切换到过期缓存，用户只在日志里看到 `[WARNING]`，但推送的消息看起来完全正常。

**建议新增：**

1. **数据质量评分卡（每日推送中新增一段）**

   今日数据质量：
   ✅ 全市场行情：新浪（5207只）
   ⚠️ 历史K线：新浪（0只东方财富）← 缺换手率

   ❌ 上证指数：API失败
   整体质量：⚠️ 3/5 — 存在数据缺失，请注意甄别
   ```

2. **数据源告警阈值**
   - 连续 2 天同一接口失败 → 微信推送"数据源持续异常"告警
   - 使用过期缓存 → 推送中显式标记
   - 情绪分降级 → 推送中说明

3. **自动降级开关**
   - 当数据质量评分低于 3/5 时，系统自动降级为"仅观察"模式，不执行实际买入确认
