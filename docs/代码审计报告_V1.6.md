### 3. `trade_review.py` — **P1 严重**

#### 3.1 [P1] `check_buy()` 中一字涨停判定逻辑有潜在误判

**文件：** `trade_review.py` 第 1022-1034 行

```python
limit_r      = 0.20 if code.startswith(("300", "301", "688")) else 0.10
at_limit_open = open_chg >= limit_r * 100 - 0.5
price_frozen  = abs(cur_price - open_p) < 0.01

unable_to_buy = False
unable_reason = ""
if at_limit_open and price_frozen:
    if vol_lots < 1000:
        unable_to_buy = True
        unable_reason = "limit_up_one_char"
    else:
        unable_reason = "possible_limit_up_unable_to_buy"
```

- **问题1：** 科创板（688开头）的涨跌幅限制是 20%，这是对的。但北交所（4/8开头）的股票也是 30% 涨跌幅，此处判别遗漏了对北交所特殊规则的处理。如果候选股中出现北交所股票，`limit_r` 会被设为 `0.10` 导致限价计算错误。
- **问题2：** `price_frozen = abs(cur_price - open_p) < 0.01`，对于高价股（比如 >200元），昨天收盘100，今天开盘110，当前110.005，正常波动但会被判为"价格冻结"。应该改用百分比阈值 `abs((cur_price - open_p) / open_p) < 0.001`。
- **影响：** 虽然短期内候选池不太可能出现北交所股票，但逻辑健壮性不足。股价冻结判定在高价股场景下可能误判。

#### 3.2 [P2] `append_rows()` 中 V1.6 计划加载在循环外部，但错误日志在内部

**文件：** `trade_review.py` 第 621-635 行

```python
_v16_flags = _load_v16_flags()
_v16_plan  = _load_v16_plan_for_report_date(report_date, _v16_flags)
```

- **问题：** V1.6 计划是一次性加载的，但如果 `_v16_plan` 是 `None`，循环内的每只股票都会打 `fallback` 标记和日志。如果当天有多只候选（6只），日志中会出现6次重复的"明日计划缺失或日期不匹配"警告。日积月累会严重污染日志。
- **建议：** 在加载后立即决定是否 fallback，出一行日志，然后让 `_apply_v16_plan_to_candidate` 静默 fallback。

#### 3.3 [P2] `_calc_row` 中有潜在除零错误

**文件：** `trade_review.py` 第 528 行

```python
ocp = round((open_p / rec_close - 1) * 100, 2)
```

- **问题：** `rec_close` 可能为 0。虽然 `_gf()` 返回 `None` 时会跳过，但如果 `rec_close` 被存储为字符串 `"0"`，`_gf` 会返回 `0.0`，`0.0` 不等于 `None`，就会进入 `rec_close != 0` 的判断——但实际上 `float("0")` 就是 `0.0`，这行代码逻辑正确但暴露了类型系统的隐患。应该加上 `rec_close > 0` 的安全判断。

### 4. `money_flow.py` — **P2 中等**

#### 4.1 [P2] 同花顺 fallback 拉全市场 20 秒，可能阻塞主流程

**文件：** `money_flow.py` 第 321-384 行

```python
df = ak.stock_fund_flow_individual(symbol=period)  # 约20秒
```

- **问题：** `_fetch_ths_snapshot()` 一次拉全市场3日排行数据约20秒。如果在 `check_buy` 的 9:36 窗口期间触发 fallback（主源挂掉），相当于在 9:36-9:37 这一分钟内阻塞 20 秒。而 9:36 买入确认窗口很窄，阻塞可能影响最后几只票的行情时效性。
- **建议：**
  1. 在 `check_money_flow_system_health()` 中，把同花顺快照的加载提前到早上（比如 08:55 跟主题龙头同批次），通过提前缓存避免盘中阻塞
  2. 或者设置更短的超时 + 快速失败

#### 4.2 [P2] `_empty_result()` 在数据缺失时仍返回 `data_source=unavailable`，但可能已被部分初始化

**文件：** `money_flow.py` 第 247-271 行

- **问题：** 多个调用点（如第 536-549 行、第 455-461 行、第 467-471 行）都用了 `_empty_result()` 做模板再 `update()`，但因为 `_empty_result()` 把 `data_source` 设为了 `unavailable`，调用方需要手动覆盖。如果任何一个调用点忘记覆盖 `data_source`，下游会错误地认为 "资金源不可用"。
- **建议：** 把 `_empty_result()` 中的 `data_source` 留空或设为 `"unknown"`，强制调用方显式设定。

#### 4.3 [P2] PROBE_SET 探针全部是大市值蓝筹，可能漏检小市值资金流问题

**文件：** `money_flow.py` 第 67-71 行

```python
PROBE_SET = ["600000", "000001", "600519", "000858", "600036"]
```

- **问题：** 东方财富 push2his 对大市值蓝筹的数据覆盖率远高于小市值。5只蓝筹全部通过不代表散户票/题材票的资金流数据也正常。如果 push2his 只在部分股票上异常（如某些板块接口挂掉），探针完全检测不到。
- **建议：** 增加一组中小市值探针（比如3-5只），或从 theme_auto 的候选池里动态选几只作为额外探针。

### 5. `data_fetcher.py` — **P1 严重**

#### 5.1 [P1] `fetch_sh_index_change()` 在接口失败时返回 `0.0`，与"上证下跌 0%"混淆

**文件：** `data_fetcher.py` 第 893-914 行

```python
def fetch_sh_index_change(trade_date: str) -> float:
    ...
    return 0.0  # 接口失败
```

- **问题：** 当 akshare 接口失败时返回 `0.0`，这和"上证指数今日收平（涨跌幅 = 0%）"在语义上无法区分。`market_guard.calc_sentiment()` 会把 `0.0` 当作正常数据使用（第 35-42 行），导致：
  - 真实市场大涨时接口失败 → 情绪分被低估
  - 真实市场大跌时接口失败 → 情绪分被高估
  - 两种场景都会影响买入决策（`market_sentiment_below_5` 判定）
- **建议：** 改为返回 `Optional[float]`，失败时返回 `None`，让调用方 `market_guard.py` 根据 `None` 做降级处理。

#### 5.2 [P2] 新浪行情在盘中大盘下跌时可能返回盘前缺省值

**文件：** `data_fetcher.py` 第 252-269 行

- **问题：** `_spot_via_sina()` 用新浪 `stock_zh_a_spot()`，该接口在盘中大盘快速下跌时可能返回盘前缓存数据。如果东方财富两个接口全挂且新浪返回的是盘前数据，全市场行情使用过期数据做选股，系统没有任何防御机制检测新浪数据的新鲜度。虽然 `_is_premarket_garbage()` 检查了 close/amount/change_pct 是否为 0（第 294-318 行），但不保证"实时数据"在盘中某个时刻被冻结。
- **影响：** 极端场景下（双接口全挂 + 新浪数据冻结），全市场快照可能滞后 30-60 分钟，选出的股票是30分钟前的"强势股"，不等于当前强势。
- **建议：** 在 `_process_spot_raw()` 中增加数据新鲜度检测（比如检查 `change_pct` 的标准差，盘前/冻结数据通常标准差极低）。

### 8. `config.yaml` 存在但 5 个潜在配置陷阱 — **P1 严重**

**文件：** `config.yaml`（已验证存在于 `stock_screener/config.yaml`）

`config.yaml` 确实存在于项目根目录（5月26日修改，4859字节），以下是配置中存在的潜在问题：

#### 8.1 [P1] `data_source.prefer_source: sina` — 生产环境默认值不合理

**文件：** `config.yaml` 第 78 行

```yaml
prefer_source: sina
```

- **问题：** 默认配置使用新浪作为首选数据源。新浪行情（`ak.stock_zh_a_spot()`）不返回换手率、量比等关键字段，且盘中大盘下跌时可能返回冻结数据。生产环境中应优先使用东方财富（auto模式），新浪仅作 fallback。
- **建议：** 改回 `prefer_source: auto`，仅在特定调试场景下手动切 sina。

#### 8.2 [P1] `apply_sell_slippage: false` 但买入时扣了滑点

**文件：** `config.yaml` 第 64 行

```yaml
apply_sell_sell_slippage: false
buy_slippage: 0.001  # 买入滑点
```

对比 `trade_review.py:1096`：
```python
adj = round(cur_price * (1 + slip), 4)  # 买入价已上浮滑点
```

- **问题：** 买入时总是扣了滑点（买入价上浮0.1%），但卖出滑点未开启。这导致"不对称滑点"：买的时候永远多付0.1%，卖的时候按实际价格。虽然策略没有严格要求对称，但这种不对称可能在统计上产生系统性偏差。

#### 8.3 [P1] `version_effective_from: "20260527"` 硬编码日期

- **问题：** V1.4 买入规则的启用日期是硬编码在配置文件里的。如果未来想复制一份配置做"策略回测对照实验"，所有阈值参数可以复制，但 `version_effective_from` 是绝对日期，不能简单复制。
- **建议：** 功能上当前设计对单线实验完全够用，未来做"全A vs 主题龙头 不同V1.4生效日"实验时会遇到此问题。

#### 8.4 [P2] 缺少 V1.5 资金条件阈值

当前 V1.5 的 `primary_inflow_days_min: 2` 和 `primary_inflow_total_min: 0` 等阈值只出现在 `config/version_flags.yaml` 中，但 `money_flow.py` 中也硬编码了这些值（`THS_HEALTHY_NET_THRESHOLD = 0.0`）。如果未来要统一调参，需要同时在两个文件中修改。应集中到 `config.yaml`。

#### 8.5 [P3] `output.directory` 和 `output.logs_directory` 被部分模块忽略

`config.yaml` 第 73-75 行定义了 `output.directory` 和 `output.logs_directory`，但 `run.py:28-30` 在代码中硬编码了路径：
```python
OUTPUT_DIR = BASE_DIR / "output"
LOGS_DIR   = BASE_DIR / "logs"
```
多个脚本（如 `trade_review.py:20`、`money_flow.py:46`）也各自用自己的路径定义。如果需要统一修改输出目录，需要改多个文件。

## 三、架构问题汇总

### 3.1 模块依赖混乱

```
run.py
  → data_fetcher.py (数据获取)
  → filters.py (筛选)
  → indicators.py (指标计算)
  → scorer.py (打分)
  → market_guard.py (情绪评分)
  → trade_review.py (审计)
  → notifier.py (推送)
  → decision_log.py / daily_report.py / excel_report.py (结果输出)
```

问题：
- 没有 `src/` 目录，所有模块 flatted 在根目录
- import 关系是隐式的（在 main() 函数中随时 import），不利于静态分析
- `theme_auto.py` 自己内部 import `data_fetcher`、`filters`、`scorer` 等，与 `run.py` 的 import 完全重复
- `money_flow.py` import `data_fetcher._raw_get`（私有函数），破坏了封装

### 3.2 命名不一致

| 位置 | 命名 | 问题 |
|------|------|------|
| `trade_review.py:28` | `COLUMNS` | 全大写常量，但 python 规范要求 `_` 前缀不表示私有 |
| `trade_review.py:72` | `SECOND_CHECK_ELIGIBLE_REASONS` | `set` 但命名像 list（复数 + s） |
| `dashboard_app.py:58-93` | `COLOR_*` | Python 中没有真正常量，但用全大写 OK 只是风格问题 |
| `run.py:26-27` | `OUTPUT_DIR` / `LOGS_DIR` | 用 `Path` 但其他模块有的用 `str` 有的用 `Path`，不一致 |

### 3.3 异常处理过于"静默"

多个模块（从 `money_flow.py` 到 `data_cache.py` 到 `notifier.py`）的异常处理模式是：

```python
try:
    ...
except Exception as e:
    logger.warning(f"...{e}")
```

这种模式在数据获取层可以接受，但在业务逻辑层（比如 `trade_review._calc_row()`）中，异常静默可能导致脏数据进入回测系统而不被发现。

## 四、数据安全与稳定性

### 4.1 Akshare 依赖风险

整个系统的数据流严重依赖 akshare 和东方财富免费接口。当前已经观察到的问题：

| 问题 | 频率 | 影响 |
|------|------|------|
| push2his 资金流接口 remote disconnect | 偶发 | 资金条件层降级或不可用 |
| 新浪行情在盘前返回占位数据 | 每日 08:00-09:25 | 被 `_is_premarket_garbage` 拦截 |
| akshare 接口限流 | 高峰时段 | 需要重试和 backoff |
| 概念板块成分股 API 挂掉 | 偶发 | theme_auto 模式退化为磁盘缓存 |

**建议：** 考虑接入一个商业数据源作为"最后一道防线"（如 Tushare Pro 的基础行情接口），仅在所有免费接口全挂时才启用。

### 4.2 本地缓存"数据毒化"风险

`data_fetcher.py` 第 708-819 行有缓存新鲜度校验（`_is_stale_hist_cache`），这是正确的做法。但 `money_flow.py` 的缓存没有任何新鲜度校验，如果在盘前误拉了一笔数据并且写入了 cache，当天剩余时间会一直使用错误数据。

## 六、按模块优先级修复建议
